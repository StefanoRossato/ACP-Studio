# 1. Introduction

## 1.1 Purpose

This specification defines the Shared Memory Protocol used by ACP Studio to exchange information between software components through a deterministic shared memory interface.

The protocol establishes a common communication contract between controller components and processing components while remaining independent from their implementation details.

---

## 1.2 Scope

This specification defines:

* the shared register space;
* register ownership;
* communication semantics;
* command and state definitions;
* protocol lifecycle;
* initialization and reset behavior;
* compliance requirements.

This specification does **not** define:

* timeout management;
* scheduling policies;
* DSP algorithms;
* application-specific behavior.

---

## 1.3 Design Goals

The protocol shall:

* provide deterministic communication;
* support asynchronous execution;
* avoid race conditions;
* define a single owner for every register;
* remain independent from implementation technologies;
* be reusable by every ACP Studio component.

---

# 2. Terminology

## 2.1 Endpoint

An **Endpoint** is a software component participating in the protocol.

Two endpoint roles are defined.

### Controller Endpoint

The Controller Endpoint initiates processing by issuing commands and reading results.

Typical implementations include Lua components.

### Processing Endpoint

The Processing Endpoint executes processing tasks and publishes processing results.

Typical implementations include JSFX components.

---

## 2.2 Register

A **Register** is a logical storage location identified by an integer index.

Registers are independent from the underlying communication mechanism.

---

## 2.3 Ownership

Each register has exactly one owner.

The owner is the only endpoint allowed to modify the register.

All other endpoints shall treat the register as read-only.

---

## 2.4 Command

A **Command** is an event generated by the Controller Endpoint.

Commands are consumed by the Processing Endpoint.

---

## 2.5 State

A **State** represents the current execution state of the Processing Endpoint.

States remain valid until replaced by a newer state.

---

# 3. Architecture

## 3.1 Overview

The protocol provides asynchronous communication between ACP Studio components through a shared register space.

```
                +----------------------+
                |  Controller Endpoint |
                |      (Lua)           |
                +----------+-----------+
                           |
                     Shared Memory
                      Register Space
                           |
                +----------+-----------+
                | Processing Endpoint  |
                |       (JSFX)         |
                +----------------------+
```

---

## 3.2 Communication Model

The protocol follows a lock-free communication model based on:

* event-driven commands;
* state-based feedback;
* single-writer ownership;
* asynchronous execution.

---

# 4. Register Space

The protocol defines a logical register space addressed by integer indices.

Register numbering is part of the protocol definition and shall remain stable across compatible protocol revisions.

Unused registers are reserved for future extensions.

---

# 5. Register Map

| Register | Symbol    | Owner      | Description                 |
| -------: | --------- | ---------- | --------------------------- |
|        0 | COMMAND   | Controller | Command register            |
|        1 | STATE     | Processing | Current processing state    |
|        2 | RMS       | Processing | RMS measurement             |
|        3 | PEAK      | Processing | Peak measurement            |
|        4 | LINEARITY | Processing | Linearity measurement       |
|        5 | SAMPLES   | Processing | Number of processed samples |
|      255 | VERSION   | Shared     | Protocol version            |

The register map defined by this specification shall not be modified by compliant implementations.

---

# 6. Register Ownership

Each register shall have exactly one owner.

Ownership rules are summarized below.

| Register  | Writer     | Reader     |
| --------- | ---------- | ---------- |
| COMMAND   | Controller | Processing |
| STATE     | Processing | Controller |
| RMS       | Processing | Controller |
| PEAK      | Processing | Controller |
| LINEARITY | Processing | Controller |
| SAMPLES   | Processing | Controller |
| VERSION   | Both       | Both       |

No register shall have multiple concurrent writers.

Endpoints shall never modify registers owned by another endpoint.

---

# 7. Commands

The COMMAND register represents an event generated by the Controller Endpoint.

| Value | Symbol |
| ----: | ------ |
|     0 | IDLE   |
|     1 | START  |
|     2 | RESET  |

After receiving a command, the Processing Endpoint shall consume the event by restoring COMMAND to IDLE.

---

# 8. States

The STATE register represents the current execution state of the Processing Endpoint.

| Value | Symbol    |
| ----: | --------- |
|     0 | IDLE      |
|     1 | MEASURING |
|     2 | COMPLETED |
|     3 | ERROR     |

Only the Processing Endpoint may modify the STATE register.

---

# 9. Protocol Lifecycle

The protocol lifecycle is defined as follows.

```
                 RESET
                   │
                   ▼
            +-------------+
            |    IDLE     |
            +-------------+
                   │
             COMMAND=START
                   │
                   ▼
          +-----------------+
          |   MEASURING     |
          +-----------------+
                   │
          Processing Complete
                   │
                   ▼
          +-----------------+
          |   COMPLETED     |
          +-----------------+
                   │
             COMMAND=RESET
                   │
                   └──────────────► IDLE
```

---

# 10. Communication Flow

A typical processing cycle is shown below.

```

        │
        ▼

COMMAND = IDLE

...

...

        ▼

```

# 11. Initialization

During initialization both endpoints shall:

1. attach the shared memory region;
2. initialize the VERSION register with the supported protocol version.

Endpoints shall not modify registers owned by other endpoints during initialization.

---

# 12. Reset

A RESET command shall produce the following observable protocol state.

| Register  | Value |
| --------- | ----: |
| COMMAND   |  IDLE |
| STATE     |  IDLE |
| RMS       |     0 |
| PEAK      |     0 |
| LINEARITY |     0 |
| SAMPLES   |     0 |
| VERSION   |     1 |

The Processing Endpoint shall consume the RESET command by restoring COMMAND to IDLE.

---

# 13. Data Validity

Measurement registers are considered valid only when:

```
STATE == COMPLETED
```

Reading measurement registers while:

* STATE == IDLE
* STATE == MEASURING

produces undefined results.

---

# 14. Versioning

Register 255 contains the protocol version.

Both endpoints shall initialize the VERSION register during initialization.

Endpoints may refuse communication when protocol versions do not match.

The protocol version defined by this specification is:

```
1
```

---

# 15. Design Principles

The protocol follows the following engineering principles:

* Single Producer / Single Consumer
* Single Writer Ownership
* Event-Based Commands
* State-Based Feedback
* Deterministic Communication
* Lock-Free Operation
* Asynchronous Execution
* Implementation Independence

---

# 16. Compliance

A compliant implementation shall:

* implement the register map exactly as defined;
* preserve register ownership;
* initialize the VERSION register;
* consume COMMAND events;
* update STATE according to this specification;
* never modify registers owned by another endpoint;
* preserve backward compatibility whenever possible.

No ACP Studio component shall redefine this protocol independently.

---

# 17. Future Extensions

Future protocol revisions may introduce:

* progress reporting;
* error codes;
* capability flags;
* performance counters;
* multiple communication channels.

Future revisions should preserve backward compatibility whenever possible.
